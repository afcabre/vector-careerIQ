## Resumen Funcional
V1 es un sistema operado por un unico tutor autenticado que trabaja sobre multiples personas consultadas. El sistema debe permitir seleccionar una persona consultada, mantener su perfil e historial, buscar y persistir vacantes, analizar ajuste, producir respuestas personalizadas y asistir en preparacion de postulaciones.

## Actores Y Roles
| Actor | Tipo | Responsabilidad |
| --- | --- | --- |
| Tutor | Humano | Acceder al sistema, seleccionar persona consultada, operar chat, busqueda y oportunidades |
| Persona consultada | Sujeto del sistema | Provee el contexto profesional que personaliza perfil, analisis y respuestas |
| LLM | Sistema externo | Generar respuestas, ayuda textual y artefactos de postulacion |
| Firestore | Sistema externo | Persistencia operacional |
| Pinecone | Sistema externo | Recuperacion semantica del CV cuando exista |
| Tavily | Sistema externo | Busqueda general, cultura organizacional y apoyo a busqueda |
| Adzuna / Remotive API | Sistemas externos | Vacantes iniciales |

## Journeys Principales
### J1. Acceso y seleccion de perfil
1. El tutor inicia sesion con credenciales preconfiguradas.
2. El sistema muestra una pantalla de seleccion de persona consultada.
3. El tutor elige una persona existente o crea una nueva.
4. El sistema carga el contexto operativo de esa persona.

### J2. Alta manual de persona consultada sin CV
1. El tutor crea una nueva persona consultada.
2. El sistema solicita como minimo `full_name`, `target_roles`, `location`, `years_experience`, `skills`.
3. El sistema guarda el perfil y permite continuar operando.
4. Si despues se agrega mas informacion, el analisis debe enriquecerse sin cambiar el flujo base.

### J3. Alta o reemplazo de CV activo
1. El tutor carga un CV para una persona consultada.
2. El sistema extrae texto y procesa indexacion vectorial.
3. El sistema asocia ese CV como unico CV activo de la persona.
4. El sistema usa ese contexto en respuestas semanticas y analisis futuros.

### J4. Conversacion continua por persona
1. El tutor entra a una persona consultada.
2. El historial previo de esa persona se recupera.
3. El tutor envia mensajes nuevos.
4. El sistema responde usando el contexto de esa persona y mantiene el hilo continuo.

### J5. Busqueda de vacantes
1. El tutor solicita busqueda para la persona consultada activa.
2. El sistema consulta proveedores iniciales.
3. El sistema normaliza resultados y presenta oportunidades.
4. Al hacer clic sobre una oportunidad encontrada, el sistema abre su detalle/consulta.
5. Solo cuando el tutor ejecute una accion explicita de guardado, la vacante se persiste como oportunidad relevante.

### J6. Ingestion manual de vacante
1. El tutor registra una vacante por URL o texto pegado.
2. El sistema extrae o normaliza contenido.
3. El sistema persiste snapshot util del contenido.
4. La vacante queda disponible para analisis y gestion operativa.

### J7. Analisis de ajuste y preparacion de postulacion
1. El tutor selecciona una vacante asociada a una persona consultada.
2. El sistema analiza ajuste perfil-vacante y fit cultural.
3. El sistema permite actualizar estado y agregar notas libres.
4. El sistema puede generar ayuda textual, carta de presentacion y resumen adaptado de experiencia.

| Modulo | Capacidades V1 |
| --- | --- |
| Autenticacion | Acceso protegido de tutor |
| Personas consultadas | Crear, listar, seleccionar, mantener perfil |

## Requisitos Funcionales
### Acceso y operacion
- `RF-01` El sistema debe autenticar al tutor antes de exponer datos sensibles.
- `RF-02` El sistema debe mostrar una pantalla inicial de seleccion de persona consultada.
- `RF-03` El tutor debe poder cambiar entre personas consultadas desde la UI.
- `RF-04` El acceso de V1 debe basarse en `username + password_hash + sesion backend + logout + rate limiting`.

### Personas consultadas y perfil
- `RF-05` El sistema debe permitir crear personas consultadas nuevas.
- `RF-06` El sistema debe permitir alta manual sin CV con al menos `full_name`, `target_roles`, `location`, `years_experience`, `skills`.
- `RF-07` El sistema debe persistir perfil estructurado por persona consultada.
- `RF-08` El sistema debe usar toda la informacion disponible del perfil cuando exista mas alla del baseline minimo.

### CV y memoria semantica
- `RF-09` El sistema debe permitir cargar CV para una persona consultada.
- `RF-10` El sistema debe mantener un solo CV activo por persona consultada en V1.
- `RF-11` Cuando exista CV, el sistema debe extraer texto, indexarlo y usarlo para respuestas semanticas y analisis.

### Conversacion
- `RF-12` El sistema debe mantener un historial conversacional persistente por persona consultada.
- `RF-13` El sistema debe recuperar ese historial cuando el tutor vuelva a abrir la persona consultada.
- `RF-14` El sistema debe mantener una sola conversacion continua por persona consultada en V1.
- `RF-15` Las respuestas del agente deben personalizarse hacia la persona consultada activa y no hacia el tutor.
- `RF-16` El sistema debe soportar respuestas en streaming.

### Busqueda e ingestion de vacantes
- `RF-17` El sistema debe consultar `Tavily`, `Adzuna` y `Remotive API` como proveedores iniciales.
- `RF-18` El sistema debe permitir registrar vacantes por busqueda automatica.
- `RF-19` Al hacer clic sobre una vacante encontrada en resultados de busqueda, el sistema debe abrir su detalle/consulta.
- `RF-20` El sistema debe persistir una vacante de busqueda automatica solo cuando el tutor ejecute una accion explicita tipo `Guardar como oportunidad`.
- `RF-21` El sistema debe permitir registrar vacantes manualmente por URL.
- `RF-22` El sistema debe permitir registrar vacantes manualmente por texto pegado.
- `RF-23` El sistema debe persistir una copia util del contenido de toda vacante registrada.
- `RF-24` El sistema solo debe persistir vacantes provenientes de busqueda automatica cuando el tutor las marque como relevantes.
- `RF-25` El sistema debe considerar dos vacantes como la misma oportunidad en V1 cuando compartan la misma URL.

### Analisis y gestion operativa
- `RF-26` El sistema debe producir analisis cualitativo de ajuste perfil-vacante.
- `RF-27` El sistema debe producir analisis cualitativo de fit cultural con evidencia publica disponible.
- `RF-28` El sistema debe permitir registrar y actualizar estados de oportunidad.
- `RF-29` Los estados validos de V1 deben ser `detected`, `analyzed`, `prioritized`, `application_prepared`, `applied`, `discarded`.
- `RF-30` El sistema debe aplicar transiciones parcialmente restringidas con secuencia principal y retroceso manual controlado.
- `RF-31` El sistema debe permitir notas operativas como texto libre breve.

### Preparacion de postulaciones
- `RF-32` El sistema debe ofrecer ayuda textual para preparar postulaciones.
- `RF-33` El sistema debe poder generar carta de presentacion.
- `RF-34` El sistema debe poder generar resumen adaptado de experiencia.
- `RF-35` El sistema debe persistir carta de presentacion y resumen adaptado como artefactos ligados a la vacante.
- `RF-36` El sistema debe conservar solo el ultimo artefacto vigente por tipo y vacante en V1.
- `RF-37` El sistema debe permitir ver y reutilizar artefactos persistidos desde el detalle de la oportunidad.

## Requisitos No Funcionales
- `RNF-01` El sistema debe mantener aislamiento de datos por `person_id`.
- `RNF-02` El sistema debe tratar acceso y datos como sensibles desde V1.
- `RNF-03` El sistema debe mantener trazabilidad minima de solicitudes, proveedor consultado y persona consultada activa.
- `RNF-04` El sistema debe degradar parcialmente si falla uno de los proveedores externos.
- `RNF-05` El streaming debe ser visible en frontend con comportamiento consistente.
- `RNF-06` La documentacion debe mantener separacion entre V1 y evolucion futura.
- `RNF-07` La arquitectura debe permitir incorporar nuevos proveedores sin reescritura mayor.
- `RNF-08` La autenticacion de V1 debe incluir sesion backend, logout y rate limiting sobre login.

## Reglas De Negocio
- `RB-01` Solo existe un operador autenticado en V1.
- `RB-02` El tutor opera en nombre de la persona consultada; no es el sujeto funcional del analisis.
- `RB-03` Cada persona consultada tiene su propio historial y no comparte conversacion con otras.
- `RB-04` Cada persona consultada tiene un solo CV activo en V1.
- `RB-05` El CV es deseable pero no obligatorio.
- `RB-06` Si no existe CV, el sistema puede operar con baseline manual minimo.
- `RB-07` El sistema debe persistir evidencia suficiente de vacantes para comparabilidad temporal.
- `RB-08` Las vacantes de busqueda automatica solo se convierten en oportunidad persistida si el tutor las marca como relevantes.
- `RB-09` En resultados de busqueda, hacer clic sobre una oportunidad solo abre su consulta/detalle.
- `RB-10` Una vacante de busqueda automatica solo se persiste mediante una accion explicita de guardado.
- `RB-11` La identidad minima de vacantes en V1 se define por `misma URL`.
- `RB-12` La secuencia principal esperada de estados es `detected -> analyzed -> prioritized -> application_prepared -> applied`.
- `RB-13` `discarded` puede alcanzarse desde cualquier estado.
- `RB-14` El sistema permite retroceso manual controlado de estado en V1.
- `RB-15` Una oportunidad en estado `discarded` puede reabrirse en V1.
- `RB-16` Las notas operativas son texto libre breve; etiquetas reutilizables quedan fuera de V1.
- `RB-17` Multi-hilo conversacional queda fuera de V1.
- `RB-18` El fit cultural debe presentarse con cautela, vacios de evidencia y sin conclusiones absolutas cuando la evidencia sea debil.
- `RB-19` Los artefactos de postulacion se asocian a una oportunidad concreta.
- `RB-20` En V1 cada oportunidad mantiene solo el ultimo artefacto vigente por tipo.
- `RB-21` El detalle de una oportunidad es un punto de acceso valido para consultar y reutilizar artefactos persistidos.
- `RB-22` Si una vacante no tiene URL confiable, V1 acepta el riesgo de duplicados.

## Casos Borde
- busqueda sin resultados relevantes
- vacante por URL inaccesible o incompleta
- vacante sin URL confiable con posible duplicado no resuelto
- texto pegado con informacion parcial o ruido
- persona consultada sin CV y con perfil manual minimo
- cambio de persona consultada mientras existe historial previo de otra
- proveedor externo caido o con respuestas parciales
- vacante despublicada despues de haber sido capturada
- CV nuevo que reemplaza al anterior como activo

## Criterios De Aceptacion
### Acceso y contexto
- `CA-01` Dado un tutor autenticado, cuando entra al sistema, entonces debe ver primero la seleccion de persona consultada.
- `CA-02` Dada una persona consultada activa, cuando el tutor conversa, entonces las respuestas deben referirse a esa persona y no al tutor.

### Perfil y CV
- `CA-03` Dado un alta manual sin CV, cuando el tutor completa el minimo util, entonces el sistema debe permitir operar con ese perfil.
- `CA-04` Dado un CV cargado, cuando finaliza su procesamiento, entonces el sistema debe considerarlo el unico CV activo del perfil.

### Conversacion
- `CA-05` Dada una persona consultada con historial previo, cuando el tutor vuelve a abrir ese perfil, entonces el historial debe recuperarse.
- `CA-06` Dadas dos personas consultadas, cuando el tutor alterna entre ellas, entonces los historiales no deben mezclarse.
- `CA-07` Dado un mensaje nuevo, cuando el backend responde en streaming, entonces el frontend debe reflejar progreso visible hasta completar.

### Vacantes
- `CA-08` Dada una busqueda automatica, cuando el tutor haga clic sobre una oportunidad en resultados, entonces el sistema debe abrir su detalle y no persistirla aun.
- `CA-09` Dada una vacante consultada desde resultados, cuando el tutor ejecute `Guardar como oportunidad`, entonces el sistema debe persistirla.
- `CA-10` Dada una vacante de busqueda automatica persistida, entonces debe conservar snapshot util del contenido.
- `CA-11` Dadas dos vacantes con la misma URL, cuando el sistema evalua identidad, entonces debe tratarlas como la misma oportunidad en V1.
- `CA-12` Dada una carga manual por URL, cuando el sistema extrae la vacante, entonces debe persistir snapshot util aun si la fuente luego desaparece.
- `CA-13` Dada una carga manual por texto pegado, cuando el tutor confirma la vacante, entonces esta debe quedar disponible para analisis y gestion.

### Analisis y postulacion
- `CA-14` Dada una vacante asociada a una persona consultada, cuando se solicita analisis, entonces el sistema debe emitir conclusion cualitativa argumentada.
- `CA-15` Dada una vacante seleccionada, cuando se prepara postulacion, entonces el sistema debe poder ofrecer ayuda textual, carta de presentacion y resumen adaptado de experiencia.
- `CA-16` Dada una vacante seleccionada, cuando se generan carta de presentacion o resumen adaptado, entonces el sistema debe persistirlos como artefactos ligados a esa vacante.
- `CA-17` Dada una vacante con un artefacto previo del mismo tipo, cuando el tutor regenere ese artefacto, entonces el sistema debe reemplazar el vigente.
- `CA-18` Dada una oportunidad con artefactos persistidos, cuando el tutor abra su detalle, entonces debe poder verlos y reutilizarlos.

### Estados
- `CA-19` Dada una oportunidad en flujo normal, cuando el tutor la avanza, entonces el sistema debe seguir la secuencia principal esperada.
- `CA-20` Dada una oportunidad en cualquier estado, cuando el tutor decide descartarla, entonces el sistema debe permitir moverla a `discarded`.
- `CA-21` Dada una oportunidad en un estado posterior, cuando el tutor necesita corregir criterio, entonces el sistema debe permitir retroceso manual controlado.
- `CA-22` Dada una oportunidad descartada, cuando el tutor decide retomarla, entonces el sistema debe permitir reabrirla hacia un estado activo.

## Estrategia De Pruebas
- pruebas unitarias para normalizacion de vacantes, validaciones y transiciones de estado
- pruebas de integracion para persistencia por `person_id`, carga de CV y consultas a proveedores
- pruebas de flujo para seleccion de persona consultada, chat continuo y analisis de vacante
- pruebas de streaming para eventos visibles y cierre correcto
- pruebas de aislamiento para evitar mezcla de historial o oportunidades entre personas

## EARS Selectivo
- Cuando el tutor seleccione una persona consultada, el sistema debera cargar su contexto operativo y conversacional.
- Si existe un CV activo para la persona consultada, el sistema debera usar recuperacion semantica en respuestas y analisis.
- Cuando una vacante sea capturada desde cualquier canal, el sistema debera persistir snapshot util de contenido.
- Si la evidencia de cultura organizacional es debil o contradictoria, el sistema debera expresarlo sin conclusiones categoricas.
