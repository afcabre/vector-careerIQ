## Resumen De Arquitectura
Arquitectura full-stack con `React + Vite` en frontend y `FastAPI` en backend. Firestore funciona como fuente operacional principal; Pinecone aporta memoria semantica del CV cuando exista. El backend orquesta autenticacion del tutor, seleccion de persona consultada, conversacion con streaming, busqueda multi-provider, persistencia de vacantes y generacion de analisis/artefactos de postulacion.

## Patron Multi-Provider
- interfaz comun por proveedor
- adaptador por API externa
- normalizacion a un esquema interno de oportunidad
- trazabilidad de origen
- tolerancia a fallo por proveedor
- degradacion parcial si un proveedor falla
- salida de busqueda con `provider_status` por ejecucion (`provider_key`, `enabled`, `attempted`, `status`, `reason`, `results_count`)
- separacion entre resultados efimeros de busqueda y oportunidades persistidas marcadas como relevantes
- identidad minima de oportunidad en V1 basada en `source_url`
- si no hay `source_url` confiable, V1 no fuerza deduplicacion secundaria
- para `Tavily` en busqueda de vacantes, la query construida se trunca a `400` caracteres maximos (limite proveedor)

## Modelo De Streaming
Recomendacion V1:
- respuesta `text/event-stream`
- endpoint principal: `POST /api/persons/{person_id}/chat/stream`
- alcance objetivo de arquitectura: aplicar streaming a todas las salidas IA relevantes (`chat`, `analyze`, `interview_brief`, `prepare`) y no solo conversacion

Eventos minimos:
- `message_start`
- `message_delta`
- `tool_status`
- `message_complete`
- `error`

Consideraciones:
- cada evento debe incluir `person_id`
- para endpoints ligados a vacantes, cada evento debe incluir `opportunity_id`
- el frontend debe poder mostrar progreso incremental
- si falla una tool externa, el stream debe poder cerrar con error controlado o degradacion parcial
- estado actual del checkpoint: streaming operativo en `chat`, `analyze`, `interview_brief` y `prepare`; en V1 se permite consumir modo no-stream cuando la accion prioriza control de consumo por cache/recalculo
- ajuste aplicado: `analyze` ahora expone SSE separado por accion (`profile-match/stream`, `cultural-fit/stream`) ademas del endpoint combinado legacy
- ajuste aplicado: `interview_brief` expone endpoint dedicado `.../interview/brief/stream` con cache por accion y persistencia de resultado

## Composicion De Prompts V1
Capas aplicadas a flujos IA:
- `guardrails_core` global unico (editable en V1)
- `system_identity` global (editable en V1)
- `task_prompt` por accion (editable en V1)
- input del usuario solo para `chat`; en `analyze/interview/prepare` se usa contexto estructurado

Principio operativo:
- por defecto se sirve ultimo resultado persistido por accion
- solo se invoca LLM cuando no hay resultado vigente o cuando el tutor marca `forzar recalculo`
- backend conserva historico por accion en `ai_action_runs`
- frontend V1 muestra ultimo resultado por defecto y habilita consulta explicita de historico por accion IA

| `GET /api/persons/{person_id}/request-traces` | backend | lectura de request payload exacto enviado a proveedores | query params opcionales: `destination`, `opportunity_id`, `run_id`, `limit` |
| `GET/PATCH /api/admin/ai-runtime-config` | backend | lectura/edicion de parametros globales de runtime IA | `top_k_semantic_analysis`, `top_k_semantic_interview`, `cv_chunking_strategy`, `cv_markdown_extraction_mode`, `interview_research_mode`, `interview_research_max_steps` |

### Politica De Ventana De Contexto Y Mensajes (OpenAI V1)
Objetivo:
- mantener respuestas personalizadas con contexto suficiente y consumo controlado
- evitar crecimiento no acotado del prompt en `chat`
- usar prompts estructurados y estables para `analyze`, `interview_brief` y `prepare`

| Flujo | Cantidad de mensajes enviados a OpenAI | Estructura del contexto | Limites actuales V1 | Regla operativa |
| --- | --- | --- | --- | --- |
| `chat` y `chat/stream` | `1 system + hasta 12` mensajes de historial (`max 13` en total) | `system = guardrails_core + system_identity + task_chat` con contexto de persona + CV; historial reciente en orden | `history max_items=12`; CV retrieval `top_k=24`; contexto CV concatenado hasta `7000` chars; fallback CV preview `1600` chars | ventana corta para conversacion continua, priorizando mensajes recientes |
| `analyze/profile-match` y `.../stream` | `2` mensajes fijos (`system`, `user`) | `system = guardrails + identity`; `user = task_analyze_profile_match` con persona, vacante y evidencia semantica | evidencia semantica de CV con `top_k` configurable (`ai_runtime_configs.top_k_semantic_analysis`, default `12`); texto de evidencia condensado para prompt | flujo analitico deterministico; no usa historial de chat |
| `analyze/cultural-fit` y `.../stream` | `2` mensajes fijos (`system`, `user`) | `system = guardrails + identity`; `user = task_analyze_cultural_fit` con persona, vacante, evidencia cultural y `confidence_hint` | Tavily cultural `max_results=4`; evidencia cultural cualitativa con vacios/alertas | separar evaluacion cultural del chat y controlar afirmaciones sin evidencia |
| `interview/brief` y `.../stream` | `2` mensajes fijos para output + planner opcional en modo adaptive | `system = guardrails + identity`; `adaptive` agrega `task_interview_research_plan`; `user final = task_interview_brief` con evidencia | Tavily entrevista por presupuesto (`ai_runtime_configs.interview_research_max_steps`) y `top_k` semantico configurable (`ai_runtime_configs.top_k_semantic_interview`, default `8`) | generar brief accionable con riesgos y preguntas soportadas por evidencia |

Notas de implementacion V1 (entrevista):
- la investigacion externa se ejecuta en ciclos multi-topico (noticias, contratacion, riesgo, sentimiento) con consultas Tavily separadas
- cada paso se traza en `request_traces` con `step_order` y metadata de tool para reconstruir el timeline del run
| `prepare` y `prepare/stream` | `2` mensajes por target seleccionado (`system`, `user`) | `system = guardrails + identity`; `user = task_prepare_*` por material (`guidance`, `cover_letter`, `experience_summary`) | evidencia semantica de CV con `top_k` configurable (`ai_runtime_configs.top_k_semantic_analysis`, default `12`); hasta `3` llamadas si se seleccionan los 3 targets | consumo controlado por `targets`; cada artefacto se genera de forma independiente |
| `analyze` combinado legacy (`/analyze`, `/analyze/stream`) | `2` mensajes fijos (`system`, `user`) | prompt unificado de perfil + cultura en una sola salida | integra evidencia semantica + cultural en un solo prompt | mantener compatibilidad mientras la UX principal usa endpoints separados |

Notas:
- estas reglas no dependen de la sesion del tutor; dependen de `person_id` y del flujo invocado
- el payload exacto de cada llamada queda persistido en `request_traces` y, cuando aplica, enlazado a `ai_action_runs` por `run_id`
- si se cambia cualquier limite, debe actualizarse esta tabla y `PROJECT_STATUS.md`

### Persistencia De Resultados IA Por Accion
- store: `ai_action_runs`
- clave funcional: `person_id + opportunity_id + action_key`
- politica V1:
  - lectura por defecto de ultimo vigente (`is_current=true`)
  - regeneracion solo con `force_recompute=true`
  - historico completo conservado en backend

## Seguridad De Arquitectura
### Requisito confirmado
- acceso protegido para el tutor
- sin correo ni OTP en V1

### Recomendacion tecnica para V1
- usuario preconfigurado del tutor
- contrasena almacenada como `password_hash`
- secretos en variables seguras
- sesion backend con cookie segura o token de sesion equivalente
- validacion de sesion antes de acceder a datos de cualquier `person_id`
- rate limiting basico sobre login
- logout explicito de sesion

### Separacion de contexto
- `operator_id` solo controla acceso
- `person_id` controla perfil, chat, oportunidades, CV y artefactos
- ninguna respuesta debe inferir identidad desde `operator_id`

## Observabilidad
- logs estructurados basicos
- `request_id` por solicitud
- incluir `person_id`, `opportunity_id` y `provider` cuando apliquen
- registrar errores de proveedor externo
- registrar cambios de estado de oportunidad
- registrar eventos clave de CV, busqueda y analisis

## Estrategia De Despliegue
### Ruta principal recomendada: Railway
- servicio separado para frontend
- servicio separado para backend
- variables seguras por servicio
- despliegue simple para V1

## Riesgos Tecnicos
- APIs externas con cobertura desigual o cambios de contrato
- scraping o extraccion incompleta desde URL
- accesos sensibles con mecanismo simple de autenticacion
- dependencia de calidad del perfil cuando no exista CV
- buffering o comportamiento variable de SSE en despliegue real
