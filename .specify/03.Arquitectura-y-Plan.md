## Resumen De Arquitectura
Arquitectura full-stack con `React + Vite` en frontend y `FastAPI` en backend. Firestore funciona como fuente operacional principal; Pinecone aporta memoria semantica del CV cuando exista. El backend orquesta autenticacion del tutor, seleccion de persona consultada, conversacion con streaming, busqueda multi-provider, persistencia de vacantes y generacion de analisis/artefactos de postulacion.

## Modelo De Streaming
Recomendacion V1:
- respuesta `text/event-stream`
- endpoint principal: `POST /api/persons/{person_id}/chat/stream`
- alcance objetivo de arquitectura: aplicar streaming a todas las salidas IA relevantes (`chat`, `analyze`, `prepare`) y no solo conversacion

Eventos minimos:
- `message_start`
- `message_delta`
- `tool_status`
- `message_complete`
- `error`

Consideraciones:
- cada evento debe incluir `person_id`
- para endpoints ligados a vacantes, cada evento debe incluir `opportunity_id`
- el frontend debe poder mostrar progreso incremental
- si falla una tool externa, el stream debe poder cerrar con error controlado o degradacion parcial
- estado actual del checkpoint: streaming operativo en `chat`, `analyze` y `prepare`; en V1 se permite consumir modo no-stream cuando la accion prioriza control de consumo por cache/recalculo
- ajuste aplicado: `analyze` ahora expone SSE separado por accion (`profile-match/stream`, `cultural-fit/stream`) ademas del endpoint combinado legacy

## Composicion De Prompts V1
Capas aplicadas a flujos IA:
- `guardrails_core` global unico (editable en V1)
- `system_identity` global (editable en V1)
- `task_prompt` por accion (editable en V1)
- input del usuario solo para `chat`; en `analyze/prepare` se usa contexto estructurado

Principio operativo:
- por defecto se sirve ultimo resultado persistido por accion
- solo se invoca LLM cuando no hay resultado vigente o cuando el tutor marca `forzar recalculo`
- backend conserva historico por accion en `ai_action_runs`
- frontend V1 muestra ultimo resultado por defecto y habilita consulta explicita de historico por accion IA

## Seguridad De Arquitectura
### Requisito confirmado
- acceso protegido para el tutor
- sin correo ni OTP en V1

### Recomendacion tecnica para V1
- usuario preconfigurado del tutor
- contrasena almacenada como `password_hash`
- secretos en variables seguras
- sesion backend con cookie segura o token de sesion equivalente
- validacion de sesion antes de acceder a datos de cualquier `person_id`
- rate limiting basico sobre login
- logout explicito de sesion

### Separacion de contexto
- `operator_id` solo controla acceso
- `person_id` controla perfil, chat, oportunidades, CV y artefactos
- ninguna respuesta debe inferir identidad desde `operator_id`
